## Context
该项目是一个全新浏览器插件，目标是把 ChatGPT 聊天记录以“可追溯 + 可分类 + 可视化”的方式同步到 GitHub。

核心挑战在于：
- 聊天内容类型不单一，必须先标准化后再分类与渲染。
- 自动分类准确率和复杂度存在不确定性，需要设计为可插拔能力。
- 自动同步需要增量游标与幂等提交，避免重复写入与历史污染。

## Goals / Non-Goals
- Goals:
  - 支持手动同步与自动增量同步。
  - 支持原始数据与分类数据双轨输出。
  - GitHub 侧可直接浏览时间线、会话和分类结果。
  - 满足日志、告警、可测试、可迭代要求。
- Non-Goals:
  - 首期不承诺二进制附件文件下载与上传（仅同步可引用元数据）。
  - 首期不强制接入外部智能分类 API。

## Decisions
- Decision: 采用分层 + 领域模型的同步流水线
  - Pipeline: `Collector -> Normalizer -> Classifier -> Renderer -> Publisher`
  - 说明: 通过职责隔离降低耦合，便于后续替换抓取和分类策略。

- Decision: 手动分类优先，自动分类为可选策略
  - 手动分类为默认且必须可用。
  - 自动分类插件返回 `label + confidence`，低于阈值时回退人工确认。
  - 说明: 满足“先可用再智能”的落地策略，降低首期复杂度和误分风险。

- Decision: 同步输出采用双轨目录
  - `raw/` 保存完整原始会话（JSON）。
  - `timeline/` 保存按时间可读的 Markdown。
  - `categorized/` 保存按角色和业务组织的内容副本。
  - `indexes/` 维护导航索引（README 与机器可读索引）。

- Decision: 自动同步使用可配置周期 + 范围约束 + 24 小时去重
  - 自动同步周期由用户配置，且必须在配置范围内。
  - 每个会话维护 `lastSyncedMessageId` 与 `lastSyncedAt`。
  - 在最近 24 小时窗口内基于消息指纹执行去重。
  - 每次发布前计算内容哈希，若无变化则跳过提交。
  - 说明: 减少重复 commit、无效 API 调用和短周期重复抓取。

- Decision: GitHub 发布鉴权支持多策略
  - 同时支持 GitHub App、OAuth、PAT。
  - 鉴权实现采用策略接口，运行时按用户配置切换。
  - 说明: 兼顾个人使用与团队集成场景。

- Decision: 业务标签采用动态模型而非固定枚举
  - 角色标签保持固定域值。
  - 业务标签基于“业务主题 + 进展情况”动态填写，不做固定枚举约束。
  - 说明: 降低标签模型对业务演进的约束。

- Decision: 发布流程默认直推目标分支
  - 不强制专用分支和 PR 流程。
  - 如后续有治理需求，可增配“分支 + PR”模式。
  - 说明: 优先保证首期链路简洁可用。

- Decision: 关键异常分级处理
  - 可降级路径失败（如自动分类失败）记录 `warn` 并继续手动流程。
  - 不可降级路径失败（如发布失败且重试耗尽）记录 `error` 并触发告警 webhook。

## Risks / Trade-offs
- ChatGPT 页面结构可能变化，抓取适配器存在维护成本。
  - Mitigation: 适配器接口化 + 快速失败日志 + 回归测试样本。
- GitHub API 频率限制会影响批量同步。
  - Mitigation: 批次提交、指数退避重试、限流配置。
- 自动分类误判会污染目录结构。
  - Mitigation: 默认关闭自动分类；启用时要求人工复核或置信度阈值。

## Migration Plan
1. 首次同步先建立目录结构和索引文件。
2. 手动同步功能先上线，验证稳定性。
3. 再启用自动同步任务和增量游标。
4. 自动分类作为后续可选模块灰度接入。
